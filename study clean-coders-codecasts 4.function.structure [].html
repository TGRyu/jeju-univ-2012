Function Structure

Overview

  • if, switch를 없애는 법
  • assignment를 최소화. 상태 변화, 사이드 이펙트를 없애기 위해
  • 함수의 많은 코드에서 에러 체크, 널 체크를 하는 것을 제거
  • 중첩된 try-catch 블럭을 갖는 함수

Arguments

  • 함수의 인자는 복잡도를 증가한다(double takes).

Three Arguments Max

  • 2개가 거의 최대이다. 드물게 3개를 사용.
  • 개인적으로 3개까지만 허용. 3개도 많지만.
  • 인자가 3개 이상이면 순서를 외우기 어렵다.
  • 3개가 넘는 인자가 함수에 전될되어야 한다면, 그 인자들은 객체인 것이 맞을 것이다.
  • 생성자의 많은 수의 인자를 넘기는 것 보다는 잘 지어진 이름을 갖는 setter가 낫다.
    • 이 경우 아주 짧은 시간 동안은 객체가 완전하지 않을 수 있다.
    • 저자에겐 많은 단위 테스트가 있어서 이런 것이 문제가 되지 않는다고.
  • 내 코드의 독자가 읽기 쉽게하고, 내 코드 자체의 위생을 위해 3개 이상의 인자는 사용하지 않는다.

no boolean arguments ever

  • boolean을 인자로 갖는 함수는 명확히 2가지 이상이 일을 하는 것이다.
    • true 경우를 위한 일 1가지와 false 경우를 위한 일 1가지
    • true 경우와 false 경우를 위해 2개의 함수를 갖는 것이 맞다.
  • 인자에 있는 boolean은 오류를 유발하거나 오해를 유발한다.
  • 2개의 boolean을 받는 함수는 4가지 일을 하는 것이다.
  • 순서는 더 복잡하다. true, false가 전달되면 무슨 의미인지 알기 어렵고, 순서가 무슨 의미를 갖는지 알기 어렵다.

Innies not Outies

  • no output arguments please
    • 사람들이 일반적으로 이렇게 생각하지 않는다.
    • argument는 함수로 전달되는 것이지, 함수로 부터 변경되어 나오는 것이라고는 생각하지 않는다.
    • output argument 대신 return value로 처리하라.

returnText

// does 'returnText' go in ? Go out ? Both ?

  • output arguments cause double takes

the null defense

  • null을 전달하거나 null을 기대하는 함수는 boolean을 전달하는 만큼 잘못된 것이다.
    • null인 경우와 행위와 null이 아닌 경우의 행위가 있다는 말이다.
    • 2개의 함수를 만드는 것이 맞다.
  • null을 pseudo boolean로 쓰지 마라.
  • defensive programming을 좋아하지 않는다.
    • 내 코드를 null, 에러 체크로 더럽히고 싶지 않다.
    • 내 동료가 실수로 null이 허용되지 않는 곳에 실수로 null을 전달하는 것을 허용하고 싶지 않다.
  • defensive programming은 잘못되었다는 단서(smell)이다.
  • defensive programming: 팀원이나 단위 테스트를 못 믿는다는 말
  • 코드에서 지속적으로 null 여부를 조사하는 것은 단위 테스트에서 이를 검증하고 있지 못하다는 의미이다.
  • public api의 경우는 defensive하게 programming한다.

the stepdown rule

  • 모든 public은 위에, 모든 private은 아래에

  • public part만 사용자들에게 전달하면 됨.
  • 예전 자바 개발자들 중에는 private 부분을 상위에 위치시키는 경우가 있었다.
  • 이 규칙과 다른 stepdown 규칙에 대해서 설명하고자 한다.
  • import part goes to top, details goes to bottom
    • 잡지의 예: 헤드라인, 시놉시스, 아웃라인, 상세한 내역들
  • 이 방식의 2가지 이점
    1. 편집자들은 마지막 부분은 필수적인 내용 전달 오류 없이 제거할 수 있다.
    2. 독자들은 제일 위에서부터 읽기 시작해서 지루해지면 그만 읽으면 된다.
  • backward reference없이 top에서 bottom으로 읽을 수 있게.

  • private 변수 선언이 위에 있지만, 이건 자바의 관행이니 걍 두자.

switches and cases

  • switch 문장 사용을 왜 꺼리나 ?
    • switch 문장은 OO 스럽지 않다 ? ← 만족할 만한 대답이 아니다.
    • OO의 가장 큰 이점 중 하나는 의존성 관리 능력이다.

  • fig-5와 같이 모듈 A와 모듈 B가 있다고 가정하자.
  • 모듈 A의 함수가 모듈 B의 함수를 호출한다고 가정하자.
  • 이 말은 모듈 A에서 모듈 B로의 의존성 관계가 있다는 말이다.
  • 이러한 의존성은 2개의 컴포넌트를 갖는다.
    • flow of control(RT 의존성): 모듈 A가 모듈 B를 호출한다.
    • source code dependency(import 등)

  • 모듈 A와 B를 독립적으로 배포하고자 한다면 어떻게 되나 ?
  • 모듈 A가 모듈 B에 source code dependency를 가지면 독립적으로 배포될 수 없다(심지어 독립적으로 컴파일될 수도 없다).
  • 모듈 B의 변경은 모듈 A의 재컴파일, 재배포를 유발한다.

  • 객체지향은 뭔가 대단한 것을 허용한다.
    • run time 의존성은 그대로 둔채로 source code 의존성을 뒤집을 수 있다.
    1. 본래의 의존성을 제거
    2. polymorphic interface를 삽입
    3. 모듈 A는 인터페이스에 의존하고, 모듈 B는 인터페이스로부터 derive한다.

  • 이렇게 되면 모듈 B의 입장에서는 source code 의존성이 역전된다.

  • B의 source code 의존성은 run time 의존성과 반대가 된다.
  • source code 의존성은 run time 의존성과 반대가 된다.
  • source code 의존성이 flow of control과 반대가 되었다.

  • 이제 모듈 A,B는 독립적으로 배포될 수 있다.
  • Independent Deployability : one of OO의 강점

  • switch 문장은 독립적 배포(Independent Deployability)에 방해가 된다.
    • 각 case 문장은 외부 모듈에 의존성을 갖는다.
    • 다수의 다른 모듈에 의존성을 갖을 수 있다.
    • fan-out problem 이라고 한다.

  • switch 문장에서 source code 의존성은 flow of control과 방향이 같다.
    • switch 문장은 모든 외부 모듈에 영향을 받는다는 것을 의미
    • 외부 모듈 중 하나라도 변경이 일어나면 switch 문장에 영향을 미치고, switch에 의존하는 다른 모든 것들에도 영향을 미친다.
    • case에서 의존하는 모듈이 변경되면 switch가 변경되고, 이를 사용하는 app가 변경되어야 한다. - redeploy되어야 한다.
    • switch 문장은 Independent Deployability를 불가능하게 하는 많은 의존성을 만든다.
  • switch 문장 제거 옵션
    1. invert switch to polymorphic dispatch
    2. move switch stmt to places that can't do any harm

  • switch 문장의 인자(타입 코드)를 abstrtact base 클래스로 치환하라.
    • switch 문장에서 수행할 코드에 대응하는 메소드를 갖는.

  • 각 case 문장은 각 case에서 수행할 작업을 구현한 derived 클래스가 된다.

  • runtime dependency(flow of control)는 그대로 이나 source code dependency가 inverted됨
  • 이제 신경 쓸 일은 언제, 어떻게 인스턴스들을 생성하는가이다.
  • 대개 팩토리에서 이런 작업을 수행한다.
    • 이게 main이라는 주요한 토픽을 가져온다.

  • 당신이 작성하는 모든 어플리케이션에 대해서 모듈 다이어그램에 코어 어플리케이션 기능을 제공하는 App와 하위 수준 상세를 제공하는 Main사이에 선을 그을 수 있어야 한다.
  • 2개의 Partition
    • Application Partition
      • 대개의 어플리케이션 코드가 존재하는 곳.
    • Main Partition
      • 하위 레벨…(factory, configuration data, main program)

  • App Partition은 다수의 서브 모듈로 나눠진다.
  • Main Partition은 should be kept small. subdivision should be limited.
  • 이 Partition들 간의 의존성은 한방향이여야하고, 한방향만 존재해야 한다.
    • Main Partition은 App Partition에 의존성을 가질 수 있지만, App Partition은 Main Partition에 의존성을 가지면 안된다.
    • in essence, Main Partition은 App Partition의 plugin이다.
  • 이러한 테크닉을 dependency injection 이라고 한다.
  • Good partitioning을 해야 하는데, FW을 과도하게 사용하여 많은 XML 파일, 설정 파일들을 만드는 개발자들도 있다.
  • dependency injection의 한가지 트릭은 caefully define and maintain your partitioning
  • FW을 사용하는 것은 좋다. 하지만 Main에서 최소한만 entry point만 App로 inject하고 Main이 factory, strategy 등을 이용하여 나머지 일을 하도록 해라.
  • Main Partition으로 내려온 switch stmt는 안전하다.

  • plugin이 자신들의 plugin은을 갖는다면 ?

  • Main 파티션과 App 파티션의 경계는 명확하나, App 파티션에서 플러그인들 간의 관계는 명확하지 않다.
  • 이 예제에서 모듈들은 고립(isolate)된 플러그인이 아니다. 각 모듈들을 배포할 때 주의를 해야 한다.
  • 모듈 A는 모듈 B의 플러그인이고, 모듈 B는 모듈 C의 플러그인이다.
  • 이제까지 설명한 Partitioning의 목적은 independently deployable 모듈로 구성된 시스템을 구성하는 것이다.
  • independently deployable == independently developable
  • switch 문장이 보여면 independently deployable 플러그인 구조로 변환해야 한다.
  • 긴 if-else 문장도 마찬가지다.

functional programming

  • 제일 먼저 창안된 패러다임.
  • 1947년. 근데 최근에 다시 유명해지고 있다.
  • assignment 없이 프로그래밍이 가능
  • 변수를 사용하는 대신, 이 변수들을 함수의 인자로 사용
  • 루프를 돌면서 변수들을 다루는 대신 재귀함수 호출을 사용
  • 값은 입력에 대해 항상 같은 결과
  • no side effect

Side Effects

public int getSize() {
    return size;
}
 
public void push(int element) {
    if(size == capacity)
        throw new Overflow();
    elements[size++] = element;
}
  • 함수가 함수보다 스코프가 큰 변수를 변경하는 경우를 생각해보자.
  • 이런 함수는 side effect를 갖는다. 이런 side effect는 다음번에 이 함수가 호출되었을 때 함수의 행위를 변경할 수 있고, 혹은 다른 함수의 행위를 변경할 수도 있다.
  • 이런 것이 함수를 읽기 어렵게 하고, 오류의 원인이 된다.
  • 종종 side effect 함수들은 짝(setter/getter, open/close, new/delete)을 이룬다. 이게 오류의 원인되는 이유이다
  • 함수들은 순서를 지켜며 호출되어야 한다. ← temporal coupling
  • temporal coupling을 지키지 않아서 오류를 겪은 적이 있을 것이다.
  • open/close처럼 쉽지만은 않다. 대부분의 temporal coupling은 감춰져있다.

  • temporal coupling을 제거할 수 있나 ?
  • 대부분의 경우 제거 가능하다. ← passing a block 기법으로 해소
  • passing a block은 다음을 보장한다.
    • consistent
    • right order
    • limited side effect

Command Query Separation

  • side effect를 관리하는 좋은 방법 중 하나.
  • command
    • 시스템의 상태 변경 가능
    • side effect를 갖는다.
    • 시스템의 상태를 변경하는 command는 아무것도 반환하지 않는다.
  • query
    • side effect가 없다.
    • 계산값이나 시스템의 상태를 반환한다.
  • 분명한 예는 getter/setter이다.
  • CQS는 다음을 형식화(formalize)한다.
    • 상태를 변경하는 함수는 값을 반환하면 안된다.
    • 값을 반환하는 함수는 상태를 변경하면 안된다.
  • 이 원칙이 이상하게 느껴질 수 있으나 분명한 이점이 있다.
int  f(); // query
 
void g(); // command
  • 함수가 side effect가 있는지 없는지를 쉽게 인식할 수 있다.
  • 다른 예를 보자.
User u = authorizer.login(userName, password);
  • command가 void가 아닌 다른 에러 코드 등을 반환하는 대신 exception을 던지도록 하는 것이 낫다.
  • 아래와 같다.
authorizer.login(userName, password);
User u = authorizer.getUser(userName);
  • 아래와 같이 상태를 변경하면서 이전 상태를 반환하는 함수를 고려해보자.
    • 저장해 놓았다가 후에 원상태로 복원하기 사용할 수 있다.
oldBlah = setBlah(newBlah);
...
setBlah(oldBlah);
  • passing a block으로 해소할 수 있다.
withBlah(newBlah, someBlahCommand);
  • 당신 코드의 독자들을 혼란스럽게 하지 말라.
    • 값을 반환하는 함수는 상태를 변경하면 안된다.
    • 상태를 변경하는 함수는 exception을 발생시킬 수는 있지만, 값을 반환할 수는 없다.

    user.execute(command);

    annuciator.promptLogin();
  • 나아가 아래와 같이 user 객체가 모든 것을 처리하도록 하는 것이 더 나아보인다.
user.execute(command, annuciator);
  • 결국 로그인되었는지 아닌지 아는 것은 user 객체이다. 이 상태는 user 객체에 속한다. 왜 user 상태를 가져다가, user를 대신해서 결정을 하는가 ? user가 해당 행위를 수행하는 것이 맞다.
  • 이게 Tell Don't Ask 예제로 연결된다.
    • tell other object what to do, but not to ask object what the state is.
    • 객체의 상태를 얻어서 객체를 대신해서 결정을 하지 말고, 객체에게 결정하도록 하는 것이 맞다.
  • 이 규칙을 준수하다보면 Q가 많이 필요없어진다. 이건 매우 좋은 것이다. 왜냐하면 Q는 곧 out of control되는 경향이 있다.
o.getX()
    .getY()
        .getZ()
            .doSomething();
  • 이런 long chain of function을 본적이 있을 것이다. ← train recks
  • Tell, Don't ask의 명확한 위반
    • 뭔가를 tell(doXXX)하기 전에 지속적으로 ask(getXXX)함으로.
  • 아래와 같이 변경하는 것이 더 좋다.
o.DoSomething();
  • o에게 무엇을 하라고 했는데, o는 어떻게 해야 할지 모른다. 하지만 o는 어떤 다른 객체에게 tell하면 되는지는 안다. 최종적으로 처리되기까지 요청이 전파된다.
  • law of demeter
    • 하나의 함수가 전체 시스템의 객체들 간의 네비게이션을 아는 것은 나쁜 아이디어이다.
  • 아래 코드가 얼마나 많은 지식이 필요한지 고려해보라.
o.getX()
    .getY()
        .getZ()
            .doSomething();
  • o는 X를 갖는다.
  • X는 Y를 갖는다.
  • Y는 Z를 갖는다.
  • Z는 doSomething() 할 수 있다.
  • 한 라인의 코드가 알아야 하는 것으로 너무 방대한 지식이고, 함수가 시스템에 너무 많이 커플링이 일어나게 한다.
  • 함수가 우리 시스템의 전체를 알게 하고 싶지는 않다. 개별 함수는 아주 제한된 지식만 가져야 한다.
  • 우리가 수행해야 할 것을 위해 이웃 객체에게 요청하길 원한다. 요청을 수신하면 적절한 객체가 수신할 때까지 전파되길 원한다.

Law of Demeter

  • law of demeter는 아래와 같은 일련의 규칙을 통해 Tell, Don't Ask를 형식화(formalize)한다.
  • You may call methods of objects that are
    1. Passed as arguments
    2. Created locally
    3. instance variables
    4. globals
  • You may NOT call methods on objects that are:
    • Returned from a previous method call.
    • o.getX().getY().getZ().doSomething()
  • 잇점: ask대신 tell하면 surrounding과 decouple된다.
  • 생물학 시스템은 tell don't ask의 좋은 예. 세포들은 다른 세포들에게 뭔가를 묻지 않는다. 대신 원하는 것을 하도록 말한다.

structured programming

  • 가장 나중에 나온 패러다임(67)
  • functional(1957), OO(62~63)
  • 3개의 패러다임이 50도 전에 10년 사이에 나왔다. 그 이후 지속되는 패러다임이 없었다.

  • 3개의 기본 오퍼레이션들(sequence, selection, iteration)로 프로그램이 구성된다.
  • single entrance at the top, single exit at the bottom의 블럭의 조합으로 프로그램을 구성

Trying is One Thing

  • 함수 내에서 try 문장이 있다면 try 문장이 변수 선언을 제외하고는 첫번째 문장이어야 한다.
  • try 블럭 내에는 한 문장(함수 호출)만 있어야 한다.
  • finally가 함수의 마지막 블럭이어야 한다. 이후에 어떤 라인도 없어야 한다.
  • 함수는 하나의 일만 해야 한다.
  • error handling은 하나의 일이다.
